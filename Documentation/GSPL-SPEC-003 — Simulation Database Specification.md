**Document Code:** GSPL-SPEC-003

**Project:** GIAR Simulation Pipeline (GSPL)

**Title:** Simulation Database Specification

**Version:** 1.0

**Status:** Draft

---

# 1. Purpose

This document defines the structure, semantics and engineering rules governing the Simulation Database generated by the GSPL pipeline.

The Simulation Database is the canonical representation of the simulation model and serves as the data exchange format between the different stages of the GSPL pipeline.

This specification defines:

- the JSON structure;
- the meaning of every section;
- the origin of every field;
- engineering constraints;
- data ownership rules.

This document is normative.

---

# 2. Scope

This specification applies to the following files:

```
2_simulation_database.json
3_simulation_database.json
4_simulation_database.json
```

Each stage of the pipeline enriches the database by adding new information while preserving the existing structure.

---

# 3. Simulation Database Philosophy

The Simulation Database represents the complete engineering description of the simulation model.

Each pipeline stage is responsible only for the information belonging to that stage.

No stage shall modify the engineering meaning of information generated by previous stages.

# 6. Metadata

The Metadata section describes the generated database.

It contains identification information only.

Example:

```json
"metadata":
{
    "format_version": "...",
    "generated_by": "...",
    "generator_version": "...",
    "generated":
    {
        "date":"...",
        "time":"..."
    },
    "project":"...",
    "author":"..."
}
```

---

# 7. Statistics

Contains summary information automatically generated by the compiler.

Typical fields include:

- number of components;
- number of objects;
- number of Shapes;
- number of Joints;
- number of Sensors;
- number of Cameras;
- number of Lights.

Statistics are informative only.

---

# 8. Components

Each engineering component is represented by one element of the Components array.

Each component contains:

```
common

frame

bounding_box

models

objects
```

# 10. Frame

Defines the reference frame of the component.

The Frame block originates from:

```
1_model_database.json
```

GSPL-02 copies this information.

GSPL-03 may enrich it with additional calculated data.

---

# 11. Bounding Box

Defines the physical volume occupied by the engineering component.

The Bounding Box contains:

- minimum corner
- maximum corner
- size

---

## GSPL-BB-001 — Component Bounding Box Source

The Bounding Box of a component shall be obtained according to the following rules.

### Case 1 — Component generated from CAD

If the component exists in the CAD model processed by GSPL-01, the Bounding Box shall be copied from:

```
1_model_database.json
```

### Case 2 — Engineering Component

If the component has been created by the engineer and has no associated CAD geometry, the Bounding Box shall be obtained from the Shape object defined in the Objects worksheet of:

```
1_assembly_table.xlsx
```

using the fields:

```
BBOX_MIN_X
BBOX_MIN_Y
BBOX_MIN_Z

BBOX_SIZE_X
BBOX_SIZE_Y
BBOX_SIZE_Z
```

---

## GSPL-BB-002 — Bounding Box Ownership

Only Shape objects define the physical volume of a component.

The following object types shall never be used to compute the component Bounding Box:

- Dummy
- Joint
- Camera
- Vision Sensor
- Proximity Sensor
- Force Sensor
- Light
- Point Cloud
- Octree
- Path

---

# 12. Models

The Models section describes the geometric representation associated with the component.

GSPL-02 creates the section.

GSPL-04 completes it after STL generation.

---

# 13. Objects

Each component owns zero or more simulation objects.

Objects belong exclusively to one component.

Objects cannot exist independently.

---

# 14. Object Structure

Each object contains:

```
common

properties
```

---

# 15. Common Block

Contains properties shared by every simulation object.

Typical fields include:

- object_id
- name
- type
- enabled
- reference_frame
- dynamic
- respondable
- visible
- collidable
- detectable
- measurable
- position
- orientation
- check
- messages
- notes

---

# 16. Properties Block

Contains parameters specific to the object type.

Examples include:

Joint

- joint_type
- cyclic
- lower_limit
- upper_limit
- motor

Camera

- camera parameters

Light

- light parameters

Dummy

- dummy parameters

Shape

- shape parameters

Each object type defines its own Properties structure.

---

# 17. Data Ownership

Every field has a unique source.

| Section | Owner |
|----------|-------|
| Metadata | GSPL-02 |
| Statistics | GSPL-02 |
| Common | Engineering Table |
| Frame | GSPL-01 |
| Bounding Box | GSPL-01 or Engineering Table |
| Models | GSPL-04 |
| Objects | Engineering Table |

---

# 18. Pipeline Evolution

The Simulation Database evolves during the pipeline.

```
GSPL-02

Engineering Database

↓

GSPL-03

Transformation Database

↓

GSPL-04

Geometry Database

↓

GSPL-05

CoppeliaSim Model
```

Each stage generates a new Simulation Database while preserving compatibility with this specification.

---

# 19. Compatibility

Future versions of the GSPL pipeline shall preserve the logical organization defined in this specification.

New sections may be added.

Existing sections shall not change their engineering meaning.
