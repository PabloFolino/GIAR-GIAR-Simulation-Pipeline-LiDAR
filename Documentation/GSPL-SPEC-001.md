# 1. Introduction

## 1.1 Purpose

This document specifies the structure and generation rules of the **CAD Database** used by the GIAR Simulation Pipeline for LiDAR (GSPL).

The CAD Database is the first engineering database generated by the pipeline and represents the complete mechanical model extracted from a Rhino CAD project.

Its purpose is to provide a deterministic, machine-readable and simulator-independent description of the CAD assembly that can be processed by the remaining stages of the GSPL.

---

## 1.2 Scope

This specification defines:

- the role of the CAD Database within the GSPL;
- the extraction rules applied to Rhino models;
- the structure of `1_model_database.json`;
- supported Rhino entities;
- geometric information extracted from every component;
- engineering metadata;
- validation rules performed by GSPL-01;
- recommended engineering practices.

This document applies to every project processed by the GSPL architecture.

---

## 1.3 Intended Audience

This specification is intended for:

- Mechanical Engineers
- Robotics Engineers
- Mechatronics Engineers
- Software Developers
- Researchers
- Contributors extending GSPL

Knowledge of Rhino is recommended.

Knowledge of the internal implementation of GSPL is not required.

---

## 1.4 CAD Database

The CAD Database is the first internal data structure generated by the GSPL.

It contains every engineering component extracted from the Rhino model together with its associated geometric information.

Unlike later databases, the CAD Database contains only information that can be obtained directly from the CAD model.

Simulation-specific concepts such as:

- joints;
- sensors;
- cameras;
- dynamics;
- simulation hierarchy;

are intentionally excluded.

Those elements are introduced during later stages of the pipeline.

---

## 1.5 Internal Pipeline File

`1_model_database.json` is an internal GSPL database.

Although the file uses the JSON format, it is not intended to be edited manually.

Instead, it is generated automatically by **GSPL-01_Rhino_Extractor.py** every time a Rhino project is processed.

Subsequent GSPL programs use this database as the authoritative source of CAD information.

---

## 1.6 Design Objectives

The CAD Database has been designed according to the following objectives.

- Preserve all relevant engineering information.
- Remain independent of any simulation platform.
- Provide deterministic output.
- Be human-readable.
- Be extensible.
- Preserve traceability back to the Rhino model.

| CAD Database | The JSON database generated by GSPL-01. |
| Rhino Object | Any entity contained in the Rhino project. |

---

## 1.9 Engineering Philosophy

The CAD model should represent only the mechanical design of the system.

Simulation concepts are intentionally separated from geometry.

This separation provides several advantages:

- CAD models remain independent of any simulator.
- Engineering decisions are introduced in later stages.
- The same CAD model can be used to generate different simulation models.
- The extraction process remains deterministic and reproducible.

This philosophy is one of the fundamental principles of the GSPL architecture.

# 2. GSPL Architecture Context

## 2.1 Overview

The **GIAR Simulation Pipeline for LiDAR (GSPL)** is organized as a sequence of independent software modules.

Each module performs one specific engineering task and generates one or more intermediate databases that become the input for the following stage.

The first stage of the pipeline is responsible for extracting engineering information from the Rhino CAD model.

The result of this process is the **CAD Database** (`1_model_database.json`), which becomes the authoritative representation of the mechanical model for the remainder of the pipeline.

---

## 2.2 GSPL Architecture

The following diagram illustrates the position of the CAD Database within the complete GSPL architecture.

```text
                    Rhino CAD Model
                           (.3dm)
                              │
                              ▼
                 GSPL-01_Rhino_Extractor.py
                              │
          ┌───────────────────┼────────────────────┐
          ▼                   ▼                    ▼
1_model_database.json   1_assembly.json   1_assembly_table.xlsx
                                                  │
                                                  ▼
                                         Simulation Engineer
                                                  │
                                                  ▼
                                   GSPL-02_Assembly_Builder.py
                                                  │
                                                  ▼
                                   2_assembly_database.json
                                                  │
                                                  ▼
                                  GSPL-03_Transform_Builder.py
                                                  │
                                                  ▼
                                  3_transform_database.json
                                                  │
                                                  ▼
                                     GSPL-04_STL_Exporter.py
                                                  │
                                                  ▼
                                           STL Components
                                                  │
                                                  ▼
                                  GSPL-05_Coppelia_Builder.py
                                                  │
                                                  ▼
                                      CoppeliaSim Model (.ttm)
```

The CAD Database is the first engineering database produced by the pipeline and serves as the foundation for every subsequent stage.

---

## 2.3 CAD Layer

The GSPL architecture separates the engineering workflow into three conceptual layers.

```text
+------------------------------------------------------+
|                    CAD Layer                         |
|------------------------------------------------------|
| Rhino CAD Model                                      |
| 1_model_database.json                                |
+------------------------------------------------------+

+------------------------------------------------------+
|                Engineering Layer                     |
|------------------------------------------------------|
| 1_assembly_table.xlsx                                |
| 1_assembly.json                                      |
| 2_assembly_database.json                             |
+------------------------------------------------------+

+------------------------------------------------------+
|                 Simulation Layer                     |
|------------------------------------------------------|
| 3_transform_database.json                            |
| STL Components                                       |
| CoppeliaSim Model                                    |
+------------------------------------------------------+
```

The purpose of this separation is to isolate mechanical information from simulation-specific information.

---

## 2.4 Responsibilities of GSPL-01

GSPL-01 performs every task related to CAD extraction.

Its responsibilities include:

- Reading the Rhino project.
- Auditing the CAD model.
- Identifying supported geometry.
- Extracting engineering components.
- Computing geometric properties.
- Computing reference frames.
- Computing Bounding Boxes.
- Generating engineering databases.

GSPL-01 does **not** generate simulation objects.

Simulation concepts such as joints, sensors, cameras and dynamics belong to later stages of the pipeline.

---

## 2.5 CAD Database as the Source of Truth

After GSPL-01 finishes, every subsequent program uses the CAD Database instead of reading the Rhino model again.

This design provides several advantages.

- Faster execution.
- Deterministic processing.
- Consistent engineering data.
- Simulator independence.
- Easier debugging.
- Complete traceability.

For this reason, `1_model_database.json` is considered the official source of geometric information throughout the GSPL.

---

## 2.6 Relationship with Other Databases

The CAD Database provides the geometric foundation for the remaining engineering databases.

```text
Rhino CAD
     │
     ▼
1_model_database.json
     │
     ├─────────────► 1_assembly.json
     │
     ├─────────────► 1_assembly_table.xlsx
     │
     ▼
2_assembly_database.json
     │
     ▼
3_transform_database.json
```

Every subsequent database enriches the information extracted from the CAD model without modifying its original geometric description.

---

## 2.7 Summary

The CAD Database represents the first structured representation of the Rhino project.

It contains only CAD-derived information and intentionally excludes simulation concepts.

This separation establishes a clear boundary between mechanical design and simulation engineering, allowing the GSPL to remain modular, deterministic and extensible.

---

# 3. Rhino CAD Model

## 3.1 Overview

The Rhino CAD model is the starting point of every GSPL project.

It represents the mechanical design of the system and serves as the only source of geometric information used by the pipeline.

The objective of GSPL-01 is to transform this CAD model into a structured engineering database while preserving the essential information required for later stages.

---

## 3.2 Engineering Philosophy

The Rhino model should describe **what the system is**, not **how it behaves**.

Consequently, the CAD model should contain only mechanical information.

Typical information includes:

- Geometry
- Dimensions
- Mechanical Components
- Layers
- Colors
- Materials
- Assembly Structure

Simulation information should not be included in the CAD model.

---

## 3.3 Supported Geometry

The current implementation supports the following Rhino object types.

| Geometry Type | Supported |
|--------------|:---------:|
| Mesh | ✔ |
| Brep | ✔ |
| Extrusion | ✔ |
| Surface | ✔ |
| SubD | Future |
| Point | Future |
| Curve | Future |

Additional object types may be incorporated in future releases.

---

## 3.4 Component Identification

Every Rhino object extracted by GSPL-01 becomes a CAD Component.

Each component receives a unique identifier generated automatically by the extractor.

Example

```text
Component ID : 17

Name : Motor_Bracket
```

The identifier remains constant throughout the pipeline and is used by every subsequent database.

---

## 3.5 Component Names

Every engineering component should have a meaningful name.

Recommended examples include:

```
Base

Motor

Camera

Lidar

Encoder

Power_Supply
```

Avoid generic names such as

```
Object01

Mesh15

Part3

Default
```

Meaningful names improve readability, traceability and engineering documentation.

---

## 3.6 Local Coordinate System

Every Rhino object possesses a local reference frame.

GSPL-01 extracts:

- Position
- Orientation

These values become the initial reference frame stored in the CAD Database.

Later stages may refine these reference frames according to engineering requirements.

---

## 3.7 Bounding Box

For every component GSPL-01 computes an axis-aligned Bounding Box.

The Bounding Box contains:

- Minimum Corner
- Maximum Corner
- Center
- Dimensions

This information is used for:

- Engineering visualization
- Geometric validation
- Transformation calculations
- STL export
- Collision estimation

---

## 3.8 Layers

Whenever available, Rhino layers are preserved.

Layer information may later be used to:

- Filter components.
- Group engineering objects.
- Define export rules.
- Generate documentation.

The current implementation stores layer information as engineering metadata.

---

## 3.9 Materials and Colors

The current version preserves material and color information when available.

These properties are considered descriptive metadata and are not currently used by the simulation pipeline.

Future versions may use this information to automatically assign visual materials inside CoppeliaSim.

---

## 3.10 CAD Model Recommendations

To obtain the best results, the Rhino project should satisfy the following recommendations.

- Every physical component should be modeled independently.
- Component names should be unique.
- Unused geometry should be removed.
- Hidden temporary objects should be deleted.
- Mechanical assemblies should be complete before extraction.
- Decorative geometry should remain separated from functional components whenever possible.

---

## 3.11 Summary

The Rhino CAD model represents the mechanical description of the system.

Its purpose is to provide a clean, structured and simulator-independent representation of the physical assembly.

GSPL-01 transforms this information into the CAD Database while preserving the geometric properties required by the remaining stages of the pipeline.

# 4. Extraction Rules

## 4.1 Overview

The primary objective of **GSPL-01_Rhino_Extractor.py** is to convert a Rhino CAD model into a deterministic engineering database.

The extraction process is intentionally designed to be independent of any simulation platform.

Every execution of GSPL-01 using the same Rhino project shall generate an identical `1_model_database.json`.

---

## 4.2 Extraction Workflow

The extraction process is composed of the following stages.

```text
Read Configuration
        │
        ▼
Open Rhino Project
        │
        ▼
Audit CAD Model
        │
        ▼
Enumerate Components
        │
        ▼
Extract Geometry
        │
        ▼
Compute Reference Frames
        │
        ▼
Compute Bounding Boxes
        │
        ▼
Generate CAD Database
```

Each stage enriches the engineering information while preserving all previously extracted data.

---

## 4.3 Component Enumeration

Every supported Rhino object becomes one engineering component.

Each component receives:

- a unique identifier;
- a unique name;
- geometric information;
- engineering metadata.

Example

```text
Rhino Object

↓

CAD Component

↓

Component ID = 17
```

The assigned identifier remains constant throughout the complete GSPL pipeline.

---

## 4.4 Supported Geometry

The extractor currently supports the following Rhino geometry.

| Geometry | Supported |
|-----------|:---------:|
| Mesh | ✔ |
| Brep | ✔ |
| Extrusion | ✔ |
| Surface | ✔ |

Objects of unsupported types are ignored and reported during the audit process.

---

## 4.5 Geometry Extraction

For every supported object the extractor retrieves:

- geometry type;
- object name;
- Rhino UUID;
- layer;
- visibility;
- material;
- color;
- local transformation;
- Bounding Box.

Whenever possible, original Rhino properties are preserved.

---

## 4.6 Reference Frame Extraction

Each component receives an initial local reference frame.

The frame contains:

- Position
- Orientation

These values are extracted directly from Rhino.

The reference frame serves as the initial engineering estimate and may later be refined by the Engineering Assembly process.

---

## 4.7 Bounding Box Computation

For every component GSPL-01 computes an axis-aligned Bounding Box.

The generated information includes:

- minimum corner;
- maximum corner;
- center;
- dimensions.

Bounding Boxes are generated automatically and are never entered manually.

---

## 4.8 Metadata Extraction

Whenever available, the following metadata are preserved.

- Rhino Object UUID
- Layer
- Material
- Color
- Visibility

Metadata improve traceability between the CAD model and the engineering databases.

---

## 4.9 Deterministic Output

The extraction process shall always produce deterministic results.

Given:

- the same Rhino project;
- the same GSPL version;
- the same configuration;

the generated CAD Database shall be identical.

This requirement is fundamental for version control and engineering traceability.

---

## 4.10 Generated Files

GSPL-01 produces three engineering artifacts.

| File | Description |
|------|-------------|
| `1_model_database.json` | CAD Database |
| `1_assembly.json` | Initial Assembly Template |
| `1_assembly_table.xlsx` | Engineering Assembly Interface |

The CAD Database is considered the primary output of the extraction process.

---

## 4.11 Summary

The extraction rules defined in this chapter guarantee that every Rhino project is converted into a deterministic and simulator-independent engineering database while preserving all relevant geometric information.

---

# 5. Specification of `1_model_database.json`

## 5.1 Overview

`1_model_database.json` is the first engineering database generated by the GSPL.

It contains the complete geometric description of every component extracted from the Rhino model.

The database intentionally excludes simulation concepts such as joints, sensors, cameras and dynamic properties.

Its only purpose is to represent the CAD model in a structured, machine-readable and simulator-independent format.

---

## 5.2 General Structure

The database is divided into two major sections:

1. Metadata
2. Components

General structure

```json
{
    "format_version":"1.0",

    "generated_by":"GSPL-01_Rhino_Extractor",

    "generator_version":"1.0.0",

    "project":"GIAR Low Cost 3D LiDAR",

    "source_file":"LIDAR_completo_ver23.3dm",

    "components":[]
}
```

---

## 5.3 Metadata

The metadata section describes the generated database.

| Field | Description |
|------|-------------|
| format_version | Database format version |
| generated_by | Program that generated the file |
| generator_version | Software version |
| project | Project name |
| source_file | Rhino source file |

Metadata allow engineering databases to be traced back to their original CAD model.

---

## 5.4 Components

The `components` array contains every engineering component extracted from Rhino.

Each Rhino object appears exactly once.

Example

```json
{
    "id":17,

    "uuid":"5f34e3d2-...",

    "name":"Motor",

    "geometry":"Mesh",

    "layer":"Mechanical",

    "visible":true,

    "material":"Steel",

    "color":[180,180,180],

    "frame":{

        "position":[0.0,0.0,0.0],

        "orientation":[0.0,0.0,0.0]

    },

    "bounding_box":{

        "min":[...],

        "max":[...],

        "center":[...],

        "size":[...]

    }

}
```

---

## 5.5 Component Fields

Each component contains the following information.

| Field | Required | Description |
|------|:--------:|-------------|
| id | ✔ | Unique component identifier |
| uuid | ✔ | Rhino UUID |
| name | ✔ | Component name |
| geometry | ✔ | Geometry type |
| layer | | Rhino layer |
| visible | ✔ | Visibility state |
| material | | Material name |
| color | | RGB color |
| frame | ✔ | Local reference frame |
| bounding_box | ✔ | Bounding Box |

---

## 5.6 Local Reference Frame

Each component contains an estimated local frame.

```json
"frame":{

    "position":[x,y,z],

    "orientation":[rx,ry,rz]

}
```

This frame represents the geometric position of the component inside the Rhino model.

Future GSPL stages may refine these values according to engineering requirements.

---

## 5.7 Bounding Box

The Bounding Box stores geometric limits.

```json
"bounding_box":{

    "min":[...],

    "max":[...],

    "center":[...],

    "size":[...]

}
```

Bounding Boxes are generated automatically and shall never be modified manually.

---

## 5.8 UUID Preservation

Every component preserves its original Rhino UUID.

This guarantees complete traceability between the CAD project and every engineering database generated by the GSPL.

---

## 5.9 Editing Policy

`1_model_database.json` is generated exclusively by GSPL-01.

It shall never be edited manually.

Engineering modifications should always be performed in the Rhino CAD model followed by a new execution of GSPL-01.

---

## 5.10 Summary

The CAD Database provides a complete, deterministic and simulator-independent description of the Rhino model.

It establishes the geometric foundation upon which every subsequent GSPL stage builds additional engineering and simulation information.

***

# 6. Geometry Representation

## 6.1 Overview

One of the primary responsibilities of **GSPL-01_Rhino_Extractor.py** is to convert the geometric information stored in the Rhino model into a structured representation suitable for engineering processing.

The objective of this representation is **not** to reproduce the Rhino model itself, but to preserve all geometric information required by the remaining stages of the GSPL.

The generated representation is therefore:

- deterministic;
- simulator independent;
- human readable;
- extensible.

---

## 6.2 Engineering Components

Every supported Rhino object becomes an Engineering Component.

Each component represents one physical element of the mechanical assembly.

Example

```text
Rhino Object
      │
      ▼
Engineering Component
```

Each Engineering Component contains:

- Identification
- Geometry
- Reference Frame
- Bounding Box
- Engineering Metadata

---

## 6.3 Geometry Types

The geometry type identifies the original Rhino object.

Current supported values include:

| Geometry Type | Description |
|--------------|-------------|
| Mesh | Polygon mesh |
| Brep | Boundary representation |
| Extrusion | Rhino extrusion |
| Surface | NURBS surface |

Future versions may support:

- SubD
- Curves
- Points
- Annotation Objects

---

## 6.4 Position Representation

Every component stores its geometric position.

```json
"frame":{

    "position":[x,y,z]

}
```

Coordinates are expressed using the project units defined in the Rhino model.

---

## 6.5 Orientation Representation

Every component stores its local orientation.

```json
"frame":{

    "orientation":[rx,ry,rz]

}
```

The orientation represents the local reference frame extracted from Rhino.

Later GSPL stages may redefine these values according to engineering requirements.

---

## 6.6 Bounding Box Representation

Every component stores an axis-aligned Bounding Box.

```json
"bounding_box":{

    "min":[x,y,z],

    "max":[x,y,z],

    "center":[x,y,z],

    "size":[dx,dy,dz]

}
```

The Bounding Box is computed automatically.

It provides:

- engineering visualization;
- collision estimation;
- dimension calculations;
- transformation support.

---

## 6.7 Colors

Whenever available, component colors are preserved.

Example

```json
"color":[

    180,

    180,

    180

]
```

Colors are considered descriptive metadata.

They are not currently used by the simulation pipeline.

---

## 6.8 Materials

Material names are preserved whenever possible.

Example

```json
"material":"Aluminum"
```

Material information may later be used for:

- rendering;
- automatic appearance generation;
- documentation.

---

## 6.9 Layers

Layer information is stored exactly as defined in Rhino.

Example

```json
"layer":"Mechanical"
```

Layers provide an additional engineering classification without affecting the simulation.

---

## 6.10 Visibility

Component visibility is preserved.

```json
"visible":true
```

Invisible objects may later be ignored by subsequent GSPL stages depending on project configuration.

---

## 6.11 UUID

Every component preserves the original Rhino UUID.

```json
"uuid":"8b5d0d1d-6d42-48fd-bd6b-..."
```

This guarantees complete traceability between the CAD model and every generated engineering database.

---

## 6.12 Geometry Independence

The CAD Database intentionally stores engineering information without introducing simulation concepts.

For example, the following information does **not** belong to this database:

- joints;
- sensors;
- cameras;
- lights;
- dynamics;
- controllers.

Those concepts belong to later engineering stages.

---

## 6.13 Summary

The geometry representation adopted by the GSPL preserves all relevant mechanical information while remaining completely independent of the target simulation platform.

This separation ensures that the CAD Database can serve as the common geometric foundation for every subsequent engineering stage.

# 7. Validation Rules

## 7.1 Overview

Before generating the CAD Database, GSPL-01 performs a series of engineering validations.

The objective of these validations is to detect CAD modeling problems as early as possible.

Whenever possible, validation errors should be reported using engineering terminology instead of software exceptions.

Each validation rule has a unique identifier.

---

## 7.2 Component Validation

### CAD-001

Every component shall have a unique identifier.

Error

```
Duplicated Component ID detected.
```

---

### CAD-002

Every component shall have a unique name.

Incorrect

```
Motor

Motor
```

Error

```
Duplicated component name.
```

---

### CAD-003

Component names shall not be empty.

Incorrect

```
Name = ""
```

---

### CAD-004

Unsupported geometry shall be reported.

Example

```
SubD
```

Warning

```
Unsupported geometry skipped.
```

---

## 7.3 Geometry Validation

### CAD-010

Every component shall contain valid geometry.

---

### CAD-011

Bounding Box dimensions shall be positive.

Incorrect

```
Size X = -25
```

---

### CAD-012

Bounding Box values shall be numeric.

---

### CAD-013

Position values shall be numeric.

---

### CAD-014

Orientation values shall be numeric.

---

### CAD-015

No coordinate shall contain NaN values.

---

### CAD-016

No coordinate shall contain infinite values.

---

## 7.4 Metadata Validation

### CAD-020

Every component shall preserve its UUID whenever available.

---

### CAD-021

Layer names shall be valid strings.

---

### CAD-022

Material names shall be valid strings.

---

### CAD-023

RGB values shall be within the interval

```
0 ≤ value ≤ 255
```

---

## 7.5 Project Validation

### CAD-030

The Rhino file shall exist.

---

### CAD-031

The Rhino file shall be readable.

---

### CAD-032

At least one supported component shall be found.

---

### CAD-033

Project units shall be identified.

---

### CAD-034

The extraction process shall complete successfully.

---

## 7.6 Validation Report

Whenever validation detects problems, GSPL-01 should generate engineering-oriented reports.

Each report should include:

- Validation Rule
- Component
- Rhino Object
- Error Description
- Suggested Action

Example

```
Validation Rule

CAD-002

Component

Motor

Error

Duplicated component name.

Suggested Action

Rename one of the duplicated Rhino objects.
```

---

## 7.7 Validation Philosophy

The objective of GSPL-01 is to detect CAD modeling problems before they propagate to the engineering stages.

Whenever possible, the extractor should continue processing the project after detecting non-critical problems.

Critical errors should stop the extraction process only when they prevent the generation of a valid CAD Database.

---

## 7.8 Summary

The validation rules defined in this chapter establish the minimum engineering quality required for a Rhino project before it enters the GSPL pipeline.

Early validation improves traceability, reduces engineering effort and simplifies debugging throughout the remaining stages of the project.

***
# 8. Best Practices

## 8.1 Overview

The quality of the CAD Database depends directly on the quality of the Rhino project.

Although GSPL-01 performs extensive validation, following the recommendations presented in this chapter will significantly improve the consistency of the generated engineering databases.

---

## 8.2 Component Modeling

### BP-001

Model every physical part as an independent Rhino object.

Avoid combining multiple mechanical components into a single CAD object.

---

### BP-002

Assign meaningful names to every component.

Recommended examples:

```
Base
Motor
Encoder
Lidar
Camera_Housing
```

Avoid names such as

```
Object001
Mesh12
Part3
```

---

### BP-003

Keep component names unique.

Duplicate names complicate engineering traceability and should always be avoided.

---

## 8.3 Geometry

### BP-004

Use clean geometry.

Delete unused construction objects before running GSPL-01.

---

### BP-005

Repair invalid meshes before extraction.

Invalid geometry may generate incorrect Bounding Boxes or export failures.

---

### BP-006

Avoid duplicated geometry occupying exactly the same position.

---

### BP-007

Maintain consistent modeling units throughout the project.

---

## 8.4 Layers

### BP-008

Organize components using meaningful Rhino layers.

Example

```
Mechanical

Electrical

Sensors

Structure
```

---

### BP-009

Avoid placing unrelated components in the same layer.

---

## 8.5 Metadata

### BP-010

Assign materials whenever possible.

Although materials are not currently used by the pipeline, they improve future compatibility.

---

### BP-011

Use colors consistently.

Colors facilitate engineering review inside Rhino.

---

## 8.6 Project Management

### BP-012

Remove obsolete geometry before extraction.

---

### BP-013

Maintain one Rhino file per engineering project.

---

### BP-014

Execute GSPL-01 after every significant CAD modification.

---

### BP-015

Do not modify `1_model_database.json` manually.

Engineering changes should always originate in the Rhino model.

---

## 8.7 Summary

Following these recommendations improves database consistency, simplifies validation and guarantees complete traceability throughout the GSPL pipeline.

---

# 10. Design Philosophy

## 10.1 Overview

The GSPL is founded on the principle that mechanical design and simulation engineering are distinct activities.

Mechanical CAD models describe **what the system is**.

Simulation models describe **how the system behaves**.

Separating these two domains produces a cleaner, more maintainable engineering workflow.

---

## 10.2 Separation of Concerns

The architecture separates three engineering layers.

```text
+--------------------------------------+
| CAD Layer                            |
|--------------------------------------|
| Rhino Model                          |
| 1_model_database.json                |
+--------------------------------------+

+--------------------------------------+
| Engineering Layer                    |
|--------------------------------------|
| 1_assembly_table.xlsx                |
| 1_assembly.json                      |
| 2_assembly_database.json             |
+--------------------------------------+

+--------------------------------------+
| Simulation Layer                     |
|--------------------------------------|
| 3_transform_database.json            |
| STL Components                       |
| CoppeliaSim Model                    |
+--------------------------------------+
```

Each layer has a single engineering responsibility.

---

## 10.3 Deterministic Engineering

Given the same Rhino project, GSPL-01 shall always generate the same CAD Database.

Deterministic processing simplifies:

- Version Control
- Debugging
- Validation
- Collaboration
- Testing

---

## 10.4 Traceability

Every engineering component can be traced back to its original Rhino object.

```text
Rhino Object

↓

CAD Component

↓

Engineering Component

↓

Simulation Object
```

This traceability is preserved throughout the complete GSPL pipeline.

---

## 10.5 Extensibility

The CAD Database has been intentionally designed to evolve without affecting the remaining engineering stages.

New metadata, geometry types and CAD importers can be added while preserving compatibility.

---

## 10.6 Simulator Independence

The CAD Database contains no information specific to CoppeliaSim.

This allows future builders to generate models for different simulation environments without modifying the CAD extraction stage.

---

## 10.7 Engineering First

The CAD model represents engineering intent.

The extractor does not reinterpret, simplify or modify the engineer's design.

Its responsibility is limited to producing an accurate digital representation of the mechanical assembly.

---

## 10.8 Final Remarks

The CAD Database is the foundation upon which every subsequent GSPL stage is built.

By providing a deterministic, simulator-independent and extensible representation of the mechanical design, GSPL-01 establishes the basis for a robust engineering workflow capable of supporting increasingly complex robotic systems.

The principles described in this specification ensure that future developments of the GSPL remain consistent with the original architectural vision while preserving compatibility with existing engineering projects.
