# GSPL-SPEC-002

# Specification of `1_assembly.json`

**GIAR Simulation Pipeline for LiDAR (GSPL)**

**Document Version:** 2.0

**Status:** Released

---

# 1. Purpose

The file `1_assembly.json` contains all engineering information required to build a simulation model in CoppeliaSim that **cannot be automatically extracted from the CAD model**.

While `1_model_database.json` describes the geometry and physical characteristics extracted from Rhino, `1_assembly.json` defines the simulation architecture, including the assembly hierarchy, simulation models and CoppeliaSim objects associated with each CAD component.

This file is automatically generated by **GSPL-01_Rhino_Extractor.py** and shall be completed by the simulation engineer before executing **GSPL-02_Assembly_Builder.py**.

---

# 2. File Structure

The file consists of two sections:

- Metadata
- Component Definitions

General structure:

```json
{
    "format_version":"1.0",

    "generated_by":"GSPL-01_Rhino_Extractor",

    "generator_version":"1.0.0",

    "project":"GIAR Low Cost 3D LiDAR",

    "root":null,

    "components":[]
}
```

---

# 3. Metadata

| Field | Description |
|--------|-------------|
| format_version | File format version. |
| generated_by | Program that generated the file. |
| generator_version | Version of the generator. |
| project | Project name. |
| root | ID of the root component. |

Example

```json
"root":1
```

---

# 4. Component Definition

Each CAD component shall appear exactly once.

Example

```json
{

    "id":18,

    "name":"NEMA17",

    "description":"Stepper motor",

    "notes":"",

    "enabled":true,

    "parent":4,

    "frame":{

        "position":[0,0,0],

        "orientation":[0,0,0]

    },

    "bounding_box":{

        "min":[...],

        "max":[...],

        "center":[...],

        "size":[...]

    },

    "models":[

        "visual"

    ],

    "objects":[

    ]

}
```

---

# 5. Component Fields

| Field | Required | Description |
|--------|:-------:|-------------|
| id | Yes | Unique component identifier generated by GSPL-01. |
| name | Yes | Component name extracted from Rhino. |
| description | No | Engineering description. |
| notes | No | Free engineering notes. |
| enabled | Yes | Enables or disables the component. |
| parent | Yes | Parent component ID. |
| frame | Yes | Component reference frame. |
| bounding_box | Yes | Component Bounding Box automatically calculated by GSPL-01. |
| models | Yes | Models where the component participates. |
| objects | Yes | List of CoppeliaSim objects created from the component. |

---

# 6. Parent

The **parent** field defines the assembly hierarchy.

Example

```json
"parent":5
```

The root component shall have

```json
"parent":null
```

The parent shall always reference another component **by its ID**.

---

# 7. Reference Frame

Every component defines a local reference frame.

Initially, GSPL-01 automatically computes the frame using the center of the component Bounding Box.

The simulation engineer may later modify this frame if another reference system better represents the mechanical behaviour.

Example

```json
"frame":{

    "position":[

        120.5,
        35.2,
        15.0

    ],

    "orientation":[

        0.0,
        0.0,
        0.0

    ]

}
```

This frame becomes the reference system for all objects belonging to the component.

---

# 8. Bounding Box

Every component stores its Bounding Box.

This information is automatically generated by GSPL-01 and shall not be manually modified.

Example

```json
"bounding_box":{

    "min":[

        110,
        20,
        10

    ],

    "max":[

        131,
        50,
        20

    ],

    "center":[

        120.5,
        35,
        15

    ],

    "size":[

        21,
        30,
        10

    ]

}
```

The Bounding Box is later used by GSPL-03 to calculate transformations, dimensions and validation rules.

---

# 9. Models

Defines the generated models where the component participates.

Supported values

| Value | Description |
|-------|-------------|
| visual | Included in the visual model. |
| simulation | Included in the simulation model. |

Examples

Visual only

```json
"models":[

    "visual"

]
```

Simulation only

```json
"models":[

    "simulation"

]
```

Both

```json
"models":[

    "visual",

    "simulation"

]
```

---

# 10. Objects

Every entry inside **objects** represents exactly one CoppeliaSim object generated from the current component.

Example

```json
"objects":[

    {

        "type":"shape",

        "name":"NEMA17",

        "position":[0,0,0],

        "orientation":[0,0,0],

        "properties":{

        }

    }

]
```

---

# 11. Common Object Structure

All CoppeliaSim objects share the same structure.

| Field | Description |
|--------|-------------|
| type | Object type. |
| name | Object name. |
| position | Local position relative to the component frame. |
| orientation | Local orientation relative to the component frame. |
| properties | Type-specific properties. |

---

# 16. Validation Rules

### RULE-001

Every component shall appear exactly once.

---

### RULE-002

Exactly one component shall be the assembly root.

---

### RULE-003

Every parent shall reference an existing component.

---

### RULE-004

The assembly shall not contain cyclic parent relationships.

---

### RULE-005

Every enabled component shall belong to at least one model.

---

### RULE-006

Every enabled component shall contain at least one object.

---

### RULE-007

Every object shall define a valid object type.

---

### RULE-008

The Bounding Box shall never be manually modified.

---

### RULE-009

Object positions are always defined relative to the component reference frame.

---

# 17. Engineering Responsibility

The simulation engineer is responsible for completing:

- Assembly hierarchy
- Parent relationships
- Description
- Engineering notes
- Simulation model assignment
- Reference frame orientation
- Additional CoppeliaSim objects
- Joint definitions
- Dummy definitions
- Sensor definitions
- Camera definitions
- Light definitions
- Any simulation-specific information

The CAD model shall never contain simulation-specific information.

---

# 18. Best Practices

## BP-001

Never modify **id**.

---

## BP-002

Never modify **name**.

Rename the Rhino object and regenerate the project instead.

---

## BP-003

Always reference parent components by **ID**.

Correct

```json
"parent":8
```

Incorrect

```json
"parent":"Base"
```

---

## BP-004

Keep the assembly hierarchy as simple as possible.

---

## BP-005

Every enabled component should normally contain at least one Shape object.

---

## BP-006

Use meaningful names.

Examples

```text
Joint_Z

Hall_Sensor

Camera_Front

LIDAR_Frame
```

---

## BP-007

Decorative components should normally belong only to the **visual** model.

---

## BP-008

Do not manually modify the Bounding Box.

---

## BP-009

Use the component frame as the origin for every object.

---

## BP-010

Before executing GSPL-02 verify:

- Root defined.
- Every enabled component has a parent.
- Parent IDs are valid.
- Models are correctly assigned.
- Required objects exist.
- Joint definitions are complete.
- No duplicated IDs exist.

---

# 19. Design Philosophy

`1_assembly.json` intentionally contains only engineering information.

All geometric information remains stored in `1_model_database.json`.

Only automatically generated reference data (Reference Frame and Bounding Box) are duplicated to simplify later stages of the pipeline.

This separation guarantees:

- Single source of truth for geometry.
- Independent simulation configuration.
- Better maintainability.
- Better traceability.
- Better scalability.

---

# 20. GSPL Philosophy

The GSPL architecture separates geometry from simulation.

Rhino defines the CAD geometry.

`1_model_database.json` describes the CAD model.

`1_assembly.json` describes the simulation architecture.

GSPL combines both sources to automatically generate the CoppeliaSim scene.

This design guarantees:

- Portability
- Maintainability
- Traceability
- Reproducibility
- Scalability

for the complete simulation pipeline.
